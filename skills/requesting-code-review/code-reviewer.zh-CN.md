## 审查清单

**代码质量：**
- 关注点分离是否清晰？
- 错误处理是否恰当？
- 类型安全（如适用）？
- 是否遵循 DRY 原则？
- 边界情况是否已处理？

**架构：**
- 设计决策是否合理？
- 是否考虑了可扩展性？
- 对性能有什么影响？
- 是否存在安全隐患？

**测试：**
- 测试是否真正在测试逻辑（而不是 mock）？
- 是否覆盖了边界情况？
- 在需要的地方是否有集成测试？
- 所有测试是否通过？

**需求：**
- 是否满足所有计划中的需求？
- 实现与规范是否匹配？
- 有没有范围蔓延？
- 是否记录了破坏性变更？

**生产就绪性：**
- 迁移策略（如有模式变更）？
- 是否考虑了向后兼容？
- 文档是否完整？
- 是否存在明显的 Bug？

[代码质量、架构或流程方面的改进]

### 评估

**可以合并吗？** [是/否/修复后可以]

**理由：** [用 1-2 句话给出技术评估]

## 关键规则

**要做：**
- 按实际严重性分类（不是所有问题都是 Critical）
- 要具体（文件:行号，而不是含糊）
- 说明问题为什么重要
- 肯定优点
- 给出明确结论

**不要：**
- 不加检查就说"看起来不错"
- 把吹毛求疵标记为 Critical
- 对你没有审查过的代码给出反馈
- 含糊其辞（"改进错误处理"）
- 回避给出明确结论
